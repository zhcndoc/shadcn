# MessageScroller — 性能

MessageScroller 默认**不进行虚拟化**。它会将每条消息都渲染为真实的 DOM
节点，并在滚动、调整大小和内容变化路径上通过几何读取来跟踪滚动位置。本文要回答的问题——不是凭信念断言，而是通过可运行的基准测试——是这种设计在较长、富含 markdown 的
对话记录中是否仍然表现良好。

**结论：是的，而且余量很大。** 在 1000 条消息上一次滚动提交会使
主线程阻塞不到 1ms；在 600 个重度 markdown 项目（64k 个 DOM 节点）上执行同样操作也远低于 1ms，且没有掉帧；而在一条重型回复逐字流式到来时，scroller 自身每个 token 的工作量约为 ~1–2ms——在 markdown 渲染之外只占用不到一帧的一小部分。极限规模下的限制因素是未虚拟化的行数本身，而不是 scroller 的工作量。

## 运行方式

```bash
# 来自 packages/react
pnpm test:browser src/message-scroller/message-scroller.perf.browser.test.tsx
```

它会通过 Playwright（`vitest.browser.config.ts`）在真实的 Chromium 中运行，并输出
每层的数值表。**它必须在真实浏览器中运行**——这里的全部成本都是强制布局，而 jsdom 会把 `getBoundingClientRect` 伪装成一次廉价的属性读取，所以在 jsdom 中做“基准测试”实际上测不到任何真实内容。该文件匹配 `*.browser.test.tsx` 的 glob，因此也会在 CI（`browser-tests.yml`）中运行，并且兼作回归防护。

**该测试套件运行在 React 的 act 环境之外。** 两个浏览器测试文件都设置了 `IS_REACT_ACT_ENVIRONMENT = false`。这个组件由事件循环驱动——`requestAnimationFrame`、`ResizeObserver`、`IntersectionObserver`——而 `act()` 无法包裹这些内容，所以测试通过 `flushSync` 来刷新它们计时的渲染，并在真实帧上完成异步滚动工作。不要在这里重新引入 `act()`：那只会再次带来由 rAF 驱动的可见性订阅者触发的、无意义的 “update not wrapped in act(...)” 警告，而不会改变实际测量结果。

## 为什么这是正确的衡量方式

滚动器的热路径开销并不是 `getBoundingClientRect` 的 _调用_ —— 而是当布局处于脏状态时，该调用触发的**强制同步重排**。这个基准测试围绕“布局何时是脏的、何时不是脏的”这两个事实构建：

- **滚动不会使布局变脏。** 修改 `scrollTop` 是滚动偏移的变化，而不是布局失效，因此在稳定滚动期间，`getBoundingClientRect` 读取的是已经计算好的盒模型。成本取决于_迭代的项目数量_（在 `getContentBottom` 中，每个顶层行读一次 rect），而几乎不受每一行子树深度的影响。这就是为什么大量 markdown 基本不会改变滚动数值。
- **增长会使布局变脏。** 当流式回复变得更高时，下一次几何读取会为一次真正的重排付费。这是 DOM 体量可能产生影响的唯一路径，因此它有自己的一层，并且与普通滚动 div 进行了 A/B 隔离。

## 三层

### 第 1 层 — 算法扩展性（隔离）

在一个真实布局完成的 DOM 上，密集循环调用 `getMessageScrollerScrollable`（它会驱动 O(n) 的 `getContentBottom` 扫描），并在每次迭代时切换 `scrollTop`，以强制触发真实滚动所带来的重排。回答的问题是：一次滚动提交会如何随行数扩展？

### 第 2 层 — 负载下的真实滚动（集成验证）

挂载一个大型对话记录，同时订阅滚动状态和可见性（一个真实应用的场景），执行从上到下的滚动，并统计 **同步滚动处理程序** 的耗时（`handleScroll → syncAfterScroll → commitScrollState → getContentBottom` 发生在 `dispatchEvent` 内，因此测量 dispatch 的耗时就是在测量用户滚动造成的、精确的主线程阻塞）。统计长任务。使用简单行内容和重型 markdown 行各运行一次。

### 第 3 层 — 流式输出（真实聊天的最坏情况）

在一段很长的对话记录底部，按 token 逐个流式生成一条助手回复，其中前面的重型轮次已经 **memoized**（因此只有不断增长的回复会重新渲染，就像真实应用一样）。在 MessageScroller（自动跟随）中和在普通滚动 `<div>`（手动跟随）中运行完全相同的增长过程；两者的 **差值** 用于隔离滚动器每个 token 的成本与 markdown 自身渲染成本。

## 说明性数字

> 在 Apple Silicon、无头 Chromium（Playwright）、2026-06 上测得。这些是  
> **说明性的，不是保证值**——绝对耗时会随硬件和 CI 而漂移。  
> 持久的约定是下面的预算断言；测试才是事实来源。  
> 重新运行以获得你机器上的数值。

**第 1 层 — 每次提交成本与行数的关系（简单行）：**

| messages | median | p95   | max   |
| -------- | ------ | ----- | ----- |
| 100      | 0.1ms  | 0.1ms | 0.7ms |
| 500      | 0.3ms  | 0.3ms | 1.2ms |
| 1000     | 0.5ms  | 0.6ms | 1.9ms |
| 2000     | 1.1ms  | 1.2ms | 4.6ms |

与行数线性增长，且常数项很小。按外推，单次提交本身直到大约 30,000 个渲染行之前都不会达到一个 16ms 帧。

**第 1 层 — 每次提交成本与 DOM 权重的关系（重型 markdown）：**

| turns | items | DOM nodes | median | p95   |
| ----- | ----- | --------- | ------ | ----- |
| 50    | 100   | 10,509    | 0.1ms  | 0.2ms |
| 150   | 300   | 31,996    | 0.2ms  | 0.3ms |
| 300   | 600   | 64,135    | 0.4ms  | 0.5ms |

64k DOM 节点几乎不会改变成本——它跟踪的是项目数量，而不是子树深度，因为滚动不会使布局失效（见上文）。

**第 2 层 — 真正的从上到下滚动：**

| scenario                    | handler median | handler p95 | handler max | long tasks |
| --------------------------- | -------------- | ----------- | ----------- | ---------- |
| 1000 trivial messages       | 0.9ms          | 1.2ms       | 2.1ms       | 0          |
| 300 heavy turns (64k nodes) | 0.6ms          | 0.8ms       | 0.9ms       | 0          |

**第 3 层 — 流式输出一条重型回复（每个 token 提交，按 `flushSync` 计时）：**

| tree                   | median     | p95   | max   | long tasks |
| ---------------------- | ---------- | ----- | ----- | ---------- |
| plain `<div>` baseline | 1.2ms      | 2.2ms | 2.3ms | 0          |
| MessageScroller        | 3.2ms      | 5.7ms | 8.3ms | 0          |
| **scroller overhead**  | **~1–2ms** |       |       |            |

每个 token 都会重新渲染不断增长的回复；`flushSync` 会对该提交以及滚动器的 `handleContentChange` 布局效果进行计时（重新扫描项目、跟随底部）。  
相较于普通 `<div>` 的额外开销，是滚动器同步的内容变更工作——稳定地 **每个 token 约 ~1–2ms**，只是一个帧的一小部分，并且没有长任务。（这一层早先使用 `act()` 包裹的版本会把整个异步 flush 计时，并错误地报告了约 0/负的额外开销；`flushSync` 的计时才是诚实的。）每个 token 的总耗时仍然主要由 markdown 渲染决定，那是消息组件的成本，而不是滚动器的成本。

## 防止回归

该测试套件会在每个触及 `packages/react/**` 的 PR 上在 CI 中运行（`.github/workflows/browser-tests.yml`）。它通过两种方式防止回归，按强度排序如下：

### 1. 确定性保护（真正有效的那个）

在一次滚动提交期间会对 `getBoundingClientRect` 进行 spy，并断言调用次数为 **O(items)，而不是 O(items²)** —— 目前大约是每个 item 1.0 次读取（2000 个 item 对应 2030 次读取）。这个计数 **与机器无关且没有方差**，因此可以设定很紧的上限：经典的性能回归（嵌套的 rect 循环、意外地对每个 item 调用 `getComputedStyle`）会使每个 item 的比率随 item 数量增长，并确定性地突破上限，而时间测试只会“无所谓”。这才是应该信任的断言。

### 2. 时间预算（粗粒度兜底）

基于墙钟时间的断言可以捕捉灾难性的变慢，但它们依赖硬件，因此需要保留很大的余量，并且只会在多倍级别的恶化时触发：

- 第 1 层：在最大行数下，每次提交的中位数 `< 16ms`（一帧）。
- 第 2 层：滚动处理器中位数 `< 16ms`，p95 `< 32ms`。
- 第 3 层：scroller 每个 token 的中位数 `< 16ms`，且相对于基线的额外开销 `< 16ms`。

不要把时间预算朝那些提示性的数值继续收紧——CI 运行器的噪声会让它们变得不稳定。如果你想更早地发现缓慢的退化，优先通过给确定性保护增加更多不变量（更多计数、在同一次运行中测量两个行数之间的缩放比例检查）来实现，而不是缩小 ms 阈值。

## 边界与非目标

- **不进行虚拟化。** 成本为 O(已渲染行数)。现实中的转写记录（数百到数千轮）具有巨大的余量；数以万计的 _实时_ 行最终会让 `getContentBottom` 的全项扫描变得重要。如果将来真的需要，修复方案已经有范围界定：最后一个非 spacer 子元素携带最大底部值，因此扫描可以收敛为 O(1)。
- **Markdown 渲染成本不在范围内。** 第 3 层表明，按 token 的成本属于内容渲染器，而不是滚动器。缓解措施（历史记录记忆化、token 批处理、代码块虚拟化）应归属于消息组件。
- **计时结果与机器相关。** 不要把这些数字当作承诺直接复制到面向用户的文档中；应注明方法并重新运行。
